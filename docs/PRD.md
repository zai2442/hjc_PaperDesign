# Product Requirements Document (PRD)

## 1. 项目背景与目标

### 1.1. 项目背景
当前多数大学校园活动管理仍依赖于传统线下或半线上模式，存在信息发布渠道分散、报名流程繁琐、数据统计效率低下、社团管理与活动脱节等问题。学生、社团负责人、辅导员及管理员在活动的发起、参与、管理和评估等环节均面临诸多不便。

### 1.2. 业务痛点
- **信息不对称**：活动信息发布不集中，学生获取信息难，错过感兴趣的活动。
- **报名效率低**：多采用群聊、在线表格等方式报名，流程不规范，易出错，且需人工审核，耗时耗力。
- **过程管理难**：活动签到多依赖人工点名，易出现代签、漏签，过程数据难以实时获取。
- **数据价值低**：活动数据（如参与度、热门活动类型等）分散且难以统计，无法为后续活动策划和学生工作评估提供有效支撑。
- **社团管理散**：社团作为活动的重要发起方，其自身的管理、成员维护、活动历史等信息缺乏系统化沉淀。

### 1.3. 预期价值与目标
本项目旨在开发一款基于 SpringBoot + Vue 的大学校园活动管理系统，通过前后端分离的现代架构，实现校园活动的全流程线上化、数字化、智能化管理。

- **对学生**：提供一个统一的活动信息门户，简化报名和签到流程，提升参与体验。
- **对社团/活动发起者**：提供便捷的活动发布、成员管理和数据监控工具，提升组织效率。
- **对辅导员/管理员**：提供高效的报名审核、数据统计与分析功能，实现对活动的有效监管和评估，为学生管理工作提供数据支持。

## 2. 功能范围与优先级

| 优先级 | 功能模块 | 功能点 | 简要描述 |
| :--- | :--- | :--- | :--- |
| **Must** | 用户管理 | 角色划分（学生、社团负责人、辅导员、超级管理员） | 系统需支持四种核心角色，并进行权限隔离。 |
| | | 用户注册与登录 | 支持统一身份认证或邮箱/学号注册。 |
| **Must** | 活动管理 | 活动发布与编辑 | 社团负责人/超级管理员可创建、编辑活动信息，包括时间、地点、详情、人数限制等。 |
| | | 活动列表与检索 | 学生可按分类、时间、热度等维度浏览和搜索活动。 |
| **Must** | 报名管理 | 学生在线报名 | 学生可一键报名参加活动，支持活动ID校验、名额扣减、重复报名拦截、状态查询。 |
| | | 报名审核 | 活动负责人/辅导员/超级管理员可对报名学生进行审核（通过/驳回）。 |
| **Must** | 签到管理 | 动态二维码签到 | 系统为每次活动生成动态二维码，用户扫码完成签到，防止作弊。 |
| | | 签到状态跟踪 | 实时记录和展示签到情况。 |
| **Should** | 数据统计 | 活动数据可视化 | 以图表形式展示活动参与人数、签到率、热门活动排行等。 |
| | | 数据导出 | 管理员可将相关统计数据导出为 Excel 报表。 |
| **Should** | 社团管理 | 社团信息展示 | 创建社团专属页面，展示社团介绍、成员、历史活动等。 |
| | | 成员管理 | 社团负责人可管理本社团成员列表。 |
| **Could** | 消息通知 | 活动状态变更通知 | 通过系统消息或邮件，向用户推送报名成功、活动临近、审核结果等通知。 |
| **Could** | 个人中心 | 我的活动日历 | 用户可在个人中心查看已报名/已参加的活动日历。 |

### 报名管理模块说明（V1.0 Must）
- **学生端**：支持在线一键报名、查看我的报名记录、取消报名；报名具备活动有效性校验、名额扣减与重复报名拦截。
- **状态枚举**：至少包含 `PENDING(待审核) / APPROVED(已通过) / REJECTED(已拒绝) / CANCELED(已取消) / COMPLETED(已完成)`。
- **管理端**：提供报名名单查看、审核流（通过/驳回、可记录驳回原因）与活动维度的数据面板（总报名数、各状态数量）。
- **质量门禁**：必须通过单元测试、接口测试与并发性能基准测试三道门禁，保证关键接口响应时间与稳定性达标。

### 迭代节奏
- **V1.0 (MVP)**: 集中实现所有 **Must** 级别的核心功能，完成活动发布 → 报名审核 → 扫码签到 → 基础信息展示的业务闭环。
- **V1.1**: 迭代 **Should** 级别的功能，重点完善数据统计和社团管理模块，提升系统的管理和运营能力。
- **V1.2**: 探索 **Could** 级别的扩展功能，如消息通知和个人中心优化，进一步提升用户体验。

## 3. 用户旅程与用例
*（详细内容见附录 `docs/PRD-appendix/user_journey.md`）*

### 3.1. 主流程：学生参与活动
1. **发现活动**：学生登录系统，在首页或活动列表页浏览到“迎新晚会”。
2. **查看详情**：点击进入活动详情页，了解时间、地点、内容。
3. **立即报名**：点击“报名”按钮，提交报名申请。
4. **等待审核**：系统显示“审核中”状态。
5. **报名成功**：收到审核通过的通知，个人中心的“我的活动”列表出现该活动。
6. **现场签到**：活动开始前，到达现场，使用手机App扫描组织者展示的动态二维码。
7. **签到成功**：系统提示“签到成功”。

### 3.2. 分支流程：活动负责人管理活动
1. **发布活动**：社团负责人登录系统，进入管理后台，填写表单发布“技术分享会”。
2. **审核报名**：活动发布后，陆续收到学生的报名申请，在“报名管理”页面对申请进行“通过”或“驳回”操作。
3. **生成签到码**：活动开始时，点击“开始签到”，系统生成一个定时刷新的动态二维码。
4. **监控数据**：活动结束后，在“数据统计”页面查看本次活动的报名人数、实际签到人数、签到率等。

### 3.3. 异常流程：
- **报名人数已满**：学生报名时，若活动人数已达上限，系统提示“报名人数已满”，报名按钮置灰。
- **重复签到**：学生扫描二维码后，若已是“已签到”状态，系统提示“请勿重复签到”。
- **二维码失效**：学生扫描时，若二维码已超过有效时间，系统提示“二维码已失效，请扫描最新二维码”。

## 4. 验收标准
*（详细测试用例见附录 `docs/PRD-appendix/test_cases.xlsx`）*

### 4.1. 验收条件示例
- **AC-FUNC-001 (活动发布)**:
  - 管理员或社团负责人成功登录后，应能看到“发布活动”的入口。
  - 发布的活动信息（名称、时间、地点、描述）必须与后台填写的内容完全一致地展示在前端列表页和详情页。
  - 未填写所有必填项时，无法成功发布。
- **AC-PERF-001 (高并发报名)**:
  - 在1000个模拟用户于10秒内同时发起报名请求的场景下，系统处理成功率应 ≥ 99%，平均响应时间 ≤ 500ms。

### 4.2. 测试用例编号规则
`[类型]-[模块]-[三位序号]`
- **类型**: `TC` (Test Case)
- **模块**: `LOGIN` (登录), `ACT` (活动), `REG` (报名), `CHK` (签到), `STAT` (统计), `CLUB` (社团)
- **示例**: `TC-ACT-001` 表示活动模块的第1个测试用例。
