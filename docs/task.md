# 一、系统设计目标

## 1. 架构设计（合理性与扩展性）

- 采用**前后端分离架构**，设计规范的 **RESTful API 接口**
- 各模块职责清晰（用户、活动、报名、签到等）
- 具备良好的**可扩展性与可维护性**，支持后续功能迭代

## 2. 核心业务流程闭环

围绕活动管理构建完整线上流程：

> **活动发布 → 用户报名 → 扫码签到 → 数据统计**

- 实现全流程数字化管理
- 确保流程顺畅、数据准确、状态可追踪

## 3. 数据模型设计

- 构建规范的数据库结构：
  - 用户表（User）
  - 活动表（Activity）
  - 报名表（Registration）
  - 签到表（Check-in）
- 明确表间关系（外键/关联）
- 保证：
  - 数据一致性
  - 数据完整性
  - 易扩展性

## 4. 用户体验与界面设计

- 前端界面设计要求：
  - 简洁美观
  - 操作便捷
  - 反馈清晰
- 提升学生用户的：
  - 使用体验（UX）
  - 视觉体验（UI）

***

# 二、系统技术难点

## 1. 高并发性能保障

在热门活动场景下（如集中报名）：

- 面临**瞬时高并发请求**
- 解决方案：
  - 数据库优化（索引、分库分表）
  - Redis缓存（热点数据缓存）
  - 消息队列（削峰填谷，如报名异步处理）
- 目标：保证系统**高响应 + 高稳定性**

***

## 2. 系统安全性设计

安全机制：

- 输入校验 + 参数化查询
- 前后端过滤
- 权限控制（RBAC）
  - 学生 / 管理员权限隔离
  - 数据访问控制

***

## 3. 扫码签到可靠性与防作弊

关键挑战：

- 动态二维码生成（带时间戳）
- 防止：
  - 重复签到
  - 代签（作弊）
- 边界问题处理：
  - 网络延迟
  - 多设备扫码

解决思路：

- 二维码短时有效（如1分钟）
- 一次性Token机制
- 签到状态幂等控制

***

## 4. 前后端协同开发与部署

作为全栈项目的工程挑战：

- 前后端并行开发
- 接口联调（API规范约定）
- 版本管理与环境统一
